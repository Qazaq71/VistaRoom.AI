# Architecture Freeze Resolution

**Статус:** Утверждено (Architecture Review завершён).
**Роль:** Chief Software Architect — финальный review после независимого Architecture Freeze Review.
**Назначение документа:** зафиксировать только окончательно принятые решения по итогам review. Это не ADR и не PCS/ACS — сами эти документы этим review не изменены; здесь перечислены только решения о том, что должно быть сделано, и кем (документация / код / новый ADR).

**Не входит в этот документ:** отклонённые замечания (PCS-001/002 boundary, отсутствие Gate у 5 capabilities) и пункты, не проверенные до конца, — они реализации не требуют.

---

## Review Scope

Architecture Review проводился исключительно по артефактам, доступным на дату проверки, в следующем составе:

- репозиторий проекта (состояние кода на дату проверки);
- PCS-001…010;
- ACS-001…010;
- ADR-000…004;
- ADR_INDEX;
- ADR_MAP;
- ADR Backlog (`ADR-Backlog-Consolidated.md`);
- Architecture Freeze Review (предшествующий независимый аудит).

Review носит статический характер: проверялось соответствие документации друг другу и фактическому наличию/отсутствию артефактов в репозитории на момент проверки.

**Review не подтверждает и не оценивает:**

- производительность;
- runtime-поведение;
- latency;
- качество результатов AI-моделей;
- бизнес-логику;
- production deployment;
- любые результаты, которые объективно можно подтвердить только после реализации.

Любые выводы за пределами этой области не входят в компетенцию данного review и не могут считаться подтверждёнными этим документом.

## Принятые решения

### R1 — Bridge между Track 1 (AI Core / ADR) и Track 2 (Roadmap v1.3 / PCS / ACS)

- **Решение:** Полный мост между всеми 20 PCS/ACS и Track 1 не строится — это было бы избыточным редизайном. Строится узкая мэппинг-таблица только для узлов, задействованных в Gate 1: Generation Intelligence (ACS-001) ↔ Generation Engine/Provider Layer (Track 1); Prompt Intelligence (ACS-004) ↔ Prompt Engine/Formatter (Track 1).
- **Где:** таблица включается как раздел внутрь ADR-006 и ADR-005 (см. R4), отдельного ADR не создаётся.
- **Владелец:** Architecture.
- **Priority:** High.

### R2 — Оригинальный документ Living Strategic Roadmap v1.3 отсутствует в репозитории

- **Решение:** Не блокирует Phase C. `docs/roadmap/README.md` дополняется пометкой, что операционным, замороженным выражением Roadmap v1.3 на сегодня являются PCS-001…010 и ACS-001…010, подтверждённые этим review. Оригинальный документ Living Strategic Roadmap v1.3 добавляется в репозиторий отдельной, не блокирующей задачей.
- **Владелец:** Governance.
- **Priority:** Medium.
- **Validation Status:** ✅ Закрыто — подтверждено Architecture Freeze Validation: `docs/roadmap/Living-Strategic-Roadmap-v1.3.md` существует в репозитории.

### R3 — Formatter: текущее состояние соответствует ожидаемому объёму Gate 1

- **Решение:** Отсутствие реализации Formatter (`src/lib/interior/prompt-engine/formatter/` содержит только README) соответствует текущему состоянию проекта и является ожидаемой частью реализации Gate 1, а не отклонением, требующим исправления перед стартом. Реализация Formatter выполняется в рамках Gate 1, после закрытия R4.
- **Владелец:** Engineering (Gate 1).
- **Priority:** High (как содержание Gate 1, не как предусловие).

### R4 — Формализация ADR-005 и ADR-006 как организационное предусловие начала Gate 1

- **Решение:** Architecture Review признаёт формализацию ADR-005 и ADR-006 единственным организационным предусловием начала Gate 1. Создаются два новых ADR-файла на основе уже полностью описанных контрактов ACS-004 и ACS-001:
  - **ADR-005** — Formatter `decisionTrace`-контракт (провизорный номер в backlog был «ADR-010»; переприсвоен, так как закрывается первым по фактическому приоритету, а не по порядку перечисления в backlog).
  - **ADR-006** — Generation Intelligence `mode`-контракт (номер из backlog подтверждён без изменений).
  - Оба включают bridge-таблицу из R1.
- **Владелец:** Architecture.
- **Priority:** Critical.
- **Blocker для Phase C Gate 1:** Да — единственное организационное предусловие.

### R5 — MY_STYLE_ID: подтверждено использование строкового литерала 'my_style'

- **Решение:** Подтверждено использование строкового литерала `'my_style'` в production-коде (`src/lib/prompts.ts:61`, `src/app/api/generate/route.ts:42`, `src/app/page.tsx:19,21,67,98`, `src/hooks/useImageGeneration.ts:81`, `src/app/components/StylePicker.tsx:6,221,222`). Review не устанавливает, является ли это незавершённой миграцией или намеренно оставленным фрагментом реализации — требуется отдельно определить это до Gate 4. `ADR-002` дополняется коротким разделом «Update», фиксирующим подтверждённое на дату review наблюдение (использование литерала) без утверждения о причине. При подтверждении, что это незавершённая миграция, — предусмотреть механический переход этих файлов на константу `MY_STYLE_ID`.
- **Владелец:** Engineering.
- **Priority:** High.
- **Blocker:** Для Gate 4 (Style Intelligence), не для Gate 1.

### R6 — Layer inversion ACS-009 ↔ PCS-009: документируется как осознанное исключение

- **Решение:** Зависимость не разворачивается (это было бы редизайном). В ACS-009 и PCS-009 добавляется по одной поясняющей строке: связь — это зависимость по данным (`customStyleDefinition`), а не функциональная/оркестрационная зависимость, и поэтому не нарушает общее правило «Level 2 использует Level 3».
- **Владелец:** Architecture (документация).
- **Priority:** Medium.

### R7 — PCS-008: формулировка блокировки уточняется

- **Решение:** Подтверждено прямой проверкой (`sourceRule` присутствует во всех ACS-004…009), что условие блокировки, сформулированное в PCS-008 («ожидание появления требования в ACS-модулях»), уже выполнено на уровне спецификации. После review причиной сохраняющейся блокировки признано отсутствие кода Domain Intelligence, а не отсутствие контракта. Формулировка в PCS-008 обновляется соответственно.
- **Владелец:** Governance (документация).
- **Priority:** Medium.

### R8 — Сведение двух ADR Backlog в один реестр

- **Решение:** Backlog из `ADR_INDEX.md` (10 неномерованных кандидатов, мир Track 1) и backlog из `ADR-Backlog-Consolidated.md` (10 кандидатов ADR-005…014, мир Track 2) сводятся в единую таблицу внутри `ADR_INDEX.md`, с пометкой источника для каждой строки. Реальной коллизии номеров не обнаружено (список `ADR_INDEX.md` не пронумерован), это задача полноты реестра, не срочного разрешения конфликта.
- **Владелец:** Governance.
- **Priority:** Medium.

### R9 — Синхронизация статусов и метаданных существующих ADR

- **Решение:** Строка `## Status` внутри файлов ADR-000, ADR-001, ADR-002, ADR-004 обновляется с «Accepted» на «Active» (соответствует фактическому использованию этих ADR другими документами). Строка ADR-004 в реестре `ADR_INDEX.md` (`Version`, `Last Updated`, History) обновляется с учётом уже существующего в файле раздела «Update — DS-7.4».
- **Владелец:** Governance.
- **Priority:** Low.

## Формальное решение по готовности к Phase C

**Gate 1 (Phase C — Integration Foundation): условно одобрен к старту.**

Единственное предусловие — закрытие **R4** (формализация ADR-005 и ADR-006). После этого Gate 1 может начинаться, включая реализацию Formatter (R3).

Пункты R1, R5, R6, R7, R8, R9 признаны реальным, подтверждённым техническим долгом, но **не блокируют** старт Gate 1. Они должны быть закрыты до соответствующих последующих Gate — в первую очередь до Gate 4 (Domain Intelligence / Style Intelligence), где R5 становится блокирующим. (R2 закрыт — см. Validation Status в R2.)
