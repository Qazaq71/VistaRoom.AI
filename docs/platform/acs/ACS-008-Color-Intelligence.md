# ACS-008 — Color Intelligence

**Статус:** AI Capability (Domain Intelligence, сервисный модуль)
**Соответствует:** Living Strategic Roadmap v1.3, Level 3 — Domain Intelligence → Color Intelligence
**Обслуживает:** Prompt Intelligence (ACS-004) как поставщик решений; закрывает открытый вопрос границы, зафиксированный в ACS-007
**Версия документа:** 1.0

---

## Purpose

Работа с цветом, палитрами, контрастом, психологией цвета — согласно формулировке Domain Intelligence из более ранних версий Roadmap. Прямо отвечает на исходную формулировку вашей цели: платформа должна "сама гармонично подбирать сочетания цвета стен, пола, мебели, аксессуаров, плитки".

## Разрешение открытого вопроса из ACS-007

В ACS-007 (Material Intelligence) было зафиксировано как нерешённое: кому принадлежат `hexToColorDescription` (конвертация HEX-кода в описательное название цвета) и правило про "белый потолок" (предотвращение перекрашивания потолка в цвет стен). Решение принимается здесь.

**Решение:**

1. **Конвертация HEX → описательное название цвета — принадлежит Color Intelligence.** Это задача описания цвета для провайдера генерации, а не задача классификации материала. Если пользователь указывает точный HEX-код предпочтения, Color Intelligence обязан преобразовать его в описательное название перед передачей в Prompt Intelligence.
2. **Правило про "белый потолок" — принадлежит Color Intelligence как явное решение, а не скрытый постфактум-фикс на уровне форматирования.** Вместо того чтобы полагаться на добавление фразы-заплатки к промпту (как это было реализовано ранее, по данным памяти о предыдущих сессиях), Color Intelligence обязан **явно принимать решение о цвете потолка** как отдельный `domainDecision` (по умолчанию — белый, если нет обоснованной причины для другого цвета), с `sourceRule`, указывающим на это правило. Это устраняет проблему на уровне источника решения, а не маскирует её на уровне текста промпта.
3. **Актуальность обеих проблем для `gpt-image-2/edit` по-прежнему не подтверждена повторно** (та же оговорка, что в ACS-004 и ACS-007) — оба механизма стоит сохранить в контракте как доступную возможность, но их фактическая необходимость (не окажется ли это решением несуществующей для новой модели проблемы) должна быть проверена тестами перед тем, как считать их обязательными в реализации.

## Responsibilities

- Формировать цветовую палитру для помещения на основе выбранного стиля (решение Style Intelligence).
- Обеспечивать гармоничное сочетание цветов между стенами, полом, мебелью, аксессуарами и плиткой — то есть согласованность **между** решениями разных поверхностей/объектов, а не определение цвета одной изолированной поверхности.
- Конвертировать явные пользовательские HEX-предпочтения в описательные названия цвета (см. решение выше).
- Принимать явное решение о цвете потолка по умолчанию (см. решение выше).
- Каждое решение сопровождать `sourceRule` (или явным `null`), аналогично требованию из ACS-004 – ACS-007.

## Non-responsibilities

Модуль **не отвечает** за:
- тип и фактуру материала — это Material Intelligence (ACS-007); **важное пограничное правило**: если материал по природе имеет собственный оттенок (например, "дуб" против "орех" — разный естественный тон), Color Intelligence не переопределяет этот оттенок самостоятельно. Возможное расхождение между палитрой, предложенной Color Intelligence, и естественным цветом материала, предложенным Material Intelligence, разрешается на уровне Rule Engine (Prompt Intelligence, ACS-004) — так же, как разрешаются любые другие конфликты между Domain Intelligence модулями, а не отдельным механизмом, специфичным для этой пары модулей;
- тип и размер мебели — Furniture Intelligence (ACS-005);
- расположение объектов — Layout & Ergonomics Intelligence (ACS-006);
- итоговую генерацию изображения — Generation Intelligence (ACS-001).

## Inputs

| Поле | Обязательность | Описание |
|---|---|---|
| `styleDecision` | Обязательно | Решение Style Intelligence, полученное через AI Orchestration. |
| `userColorPreference` | Опционально | Явное указание пользователя, включая возможный HEX-код. |
| `spaceType` | Обязательно | Тип пространства. |

## Outputs

| Поле | Описание |
|---|---|
| `colorPalette` | Согласованный набор цветов для стен, пола, мебели, аксессуаров, плитки, с `sourceRule` для каждого элемента. |
| `ceilingColorDecision` | Явное решение о цвете потолка (см. разрешение открытого вопроса выше). |
| `resolvedColorDescription` | Описательное название цвета, если на входе был HEX-код. |

## Dependencies

- **Восходящие:** AI Orchestration (ACS-003) — источник `styleDecision`.
- **Нисходящие (потребители):** Prompt Intelligence (ACS-004) — получает `colorPalette` и `ceilingColorDecision` как часть `domainDecisions`; конфликты с Material Intelligence разрешаются там же, через Rule Engine.

**Запрещённые зависимости:** те же ограничения, что в ACS-005/006/007 — не вызывает Provider Layer, Prompt Intelligence, другие Domain Intelligence модули напрямую, включая Material Intelligence (разрешение конфликтов — исключительно через Rule Engine, ACS-004).

## Consumers

Prompt Intelligence (ACS-004).

## Extension Points

- Расширение палитры на психологию цвета (упомянуто в Purpose, но не детализировано в Responsibilities этого документа — требует отдельной проработки, если будет подтверждена продуктовая ценность этого направления).
- Пересмотр необходимости `resolveHexToDescription` и `ceilingColorDecision` как обязательных механизмов — по итогам проверки актуальности для `gpt-image-2/edit` (см. решение выше, пункт 3).

## Knowledge Packs

**Residential Colors** и **Commercial Colors** — по аналогии с принципом Commercial Dimension (точное имя не зафиксировано в Roadmap v1.3 напрямую, как и для Material Intelligence в ACS-007 — требует подтверждения при реализации).

## Tests

- **Контрактные тесты:** каждое решение сопровождается `sourceRule` или явным `null`; `ceilingColorDecision` присутствует всегда, даже если совпадает со значением по умолчанию (белый).
- **Boundary-тесты:** модуль не вызывает Material Intelligence, Prompt Intelligence, Provider Layer напрямую.
- **Регрессионный тест на конвертацию HEX:** заданный набор тестовых HEX-значений (включая случаи из более ранней истории проекта — бежевый, терракотовый) корректно конвертируется в описательные названия.
- **Тест на актуальность для текущей модели** (см. пункт 3 разрешения открытого вопроса): необходимо подтвердить экспериментально, воспроизводится ли проблема "потолок перекрашивается в цвет стен" на `gpt-image-2/edit` — если нет, `ceilingColorDecision` можно упростить до менее навязчивого решения (но не убирать полностью, так как явное решение о цвете потолка полезно само по себе для согласованности палитры).
