# ACS-003 — AI Orchestration

**Статус:** AI Capability (сервисный модуль)
**Соответствует:** Living Strategic Roadmap v1.3, Level 3 — AI Orchestration
**Обслуживает:** PCS-003, PCS-004, а также координирует все Domain Intelligence модули для PCS-001/002
**Версия документа:** 1.0

---

## Purpose

Координировать вызовы остальных AI-модулей: определять, какие агенты нужны, в каком порядке, какие знания использовать. Состоит из трёх компонентов по Roadmap v1.3: AI Orchestrator (принятие решений о последовательности), Context Composer (сборка единого контекста), Agent Framework (общий контракт для AI-модулей).

Это модуль, в рамках которого — по вашему явному указанию в предыдущем сообщении — должно быть принято архитектурное решение о параллельной или последовательной обработке нескольких помещений в проекте (отложенный вопрос из PCS-003).

## Responsibilities

- Определять порядок вызова Domain Intelligence модулей для одного помещения (например, Style Intelligence перед Material Intelligence, если материал зависит от выбранного стиля).
- Формировать единый контекст (Context Composer) из Spatial, Style, Knowledge, User, Furniture, Materials — согласно Level 3 Roadmap v1.3.
- Предоставлять единый контракт (Agent Framework), которому подчиняются все AI-модули — единообразные методы вроде `analyze()`, `plan()`, `review()`, `suggest()` (терминология из более ранней версии Roadmap, требует подтверждения при реализации).
- **Принимать решение о модели обработки нескольких помещений внутри одного проекта** (см. раздел "Архитектурное решение" ниже) — это прямая зона ответственности данного модуля, а не Project Intelligence (PCS-003), которая только использует результат.

## Non-responsibilities

Модуль **не отвечает** за:
- содержание самих решений (что за стиль, что за материал) — это Domain Intelligence;
- вызов провайдера генерации — это Generation Intelligence (ACS-001);
- анализ фото и формирование Structured Scene — это Scene Intelligence (ACS-002);
- хранение пользовательских данных между сессиями — это Personal AI Designer (PCS-009).

## Архитектурное решение: обработка нескольких помещений проекта

Ранее в PCS-003 и PCS-004 этот вопрос был зафиксирован как Decision Deferred. Согласно вашему указанию, решение принимается здесь.

**Решение: параллельная обработка на уровне генерации, с предварительным последовательным разрешением контекста стиля на уровне проекта.**

Обоснование:

1. **Согласованность стиля между помещениями (Cross-room Style Consistency, требование PCS-003) не требует, чтобы генерация одной комнаты технически зависела от результата генерации другой.** Согласованность обеспечивается тем, что Context Composer формирует единый **Project Design Context** (выбранный стиль, палитра, материалы) **один раз, до начала генерации любого помещения**, а затем этот же контекст передаётся в промпт для каждого помещения независимо. То есть источник согласованности — общий входной контекст, а не последовательная передача результата от комнаты к комнате.
2. При таком подходе **вызовы Generation Intelligence для разных помещений одного проекта не имеют взаимной зависимости** и могут выполняться параллельно без потери согласованности.
3. **Ограничение, которое нужно учитывать при реализации:** по данным более раннего архитектурного аудита проекта, тарифные уровни (`profi`/`agency`) на момент аудита не были назначены, и все запросы ограничивались как `free` — если это ограничение всё ещё актуально на момент реализации Gate 6, параллельные вызовы могут упереться в rate-limit провайдера или платформы раньше, чем в стоимость или логическую консистентность. Это не отменяет архитектурное решение о параллелизме, но означает, что **степень параллелизма (сколько помещений обрабатывать одновременно) должна быть настраиваемым параметром**, а не жёстко зафиксированным числом, и должна быть согласована с фактическим состоянием rate-limiting на момент реализации.
4. Последовательная обработка не исключается полностью: если в будущем появится требование, чтобы более поздние помещения учитывали *визуальные детали* уже сгенерированных (а не только заранее выбранный стиль/палитру) — например, точное совпадение оттенка уже выбранного материала по факту генерации, а не по описанию — потребуется пересмотр этого решения через Decision Governance, так как это изменит модель на последовательную с передачей результата между вызовами.

**Это решение фиксируется как принятое, а не как временная оценка** — в отличие от Performance Budget других модулей в этой серии документов, где отсутствие данных явно помечалось как "не подтверждено". Здесь решение является архитектурным выбором, а не эмпирическим фактом, поэтому не требует "подтверждения" в том же смысле — оно может быть пересмотрено только через формальный процесс Decision Governance, а не по мере поступления новых данных.

## Архитектурное решение: порядок вызова Domain Intelligence модулей

Это отдельный уровень параллелизма — не путать с решением выше (которое касается обработки нескольких **помещений** проекта). Здесь речь о порядке вызова Domain Intelligence модулей **внутри одного** помещения. Вопрос был отмечен как пробел при составлении ACS-009 (Style Intelligence) и явно поручен к решению здесь.

**Решение основано на фактических зависимостях, зафиксированных в уже принятых ACS-005 – ACS-009, а не выбрано произвольно:**

- **Style Intelligence (ACS-009)** не имеет входных зависимостей от других Domain Intelligence модулей — выполняется **первым, последовательно**, до вызова остальных.
- **Furniture Intelligence (ACS-005), Material Intelligence (ACS-007), Color Intelligence (ACS-008)** — каждый из них принимает `styleDecision` как обязательный вход, но не зависит друг от друга. После завершения Style Intelligence эти три модуля могут вызываться **параллельно**.
- **Layout & Ergonomics Intelligence (ACS-006)** принимает `furnitureRecommendation[]` как обязательный вход (согласно контракту ACS-006) — то есть зависит от результата Furniture Intelligence и должен вызываться **после** него, а не параллельно с ним. При этом Layout может выполняться параллельно с Material и Color Intelligence, так как от них не зависит.

Таким образом, полный порядок для одного помещения:

```
Style Intelligence (последовательно, первым)
        ↓
┌───────────────────┬──────────────────────┬────────────────────┐
Furniture Intelligence   Material Intelligence   Color Intelligence
        ↓                    (параллельно)          (параллельно)
Layout & Ergonomics
Intelligence
        ↓
   (все решения собраны → Prompt Intelligence, ACS-004)
```

**Важная оговорка:** этот порядок закрывает вопрос последовательности вызовов, но сам по себе не устраняет ситуацию, когда Layout обнаруживает нарушение эргономики уже после того, как Furniture Intelligence отработал. Решение этой ситуации — в следующем разделе.

## Архитектурное решение: разрешение конфликта Layout ↔ Furniture Intelligence

Ранее зафиксировано как открытый вопрос в ACS-006 и ACS-010. Решение принимается здесь, так как оно затрагивает именно порядок и повторность вызовов — прямую зону ответственности AI Orchestration.

**Решение: ограниченный цикл повторного согласования на уровне оркестрации, до вызова Prompt Intelligence.**

Механизм:

1. Furniture Intelligence и Layout & Ergonomics Intelligence вызываются в порядке, зафиксированном выше.
2. Если Layout возвращает `ergonomicIssue`, каждая запись обязана содержать ссылку на конкретный предмет мебели, вызвавший нарушение (`causingItemId`) — это новое требование к выходным данным ACS-006, добавленное этим решением (см. обновление ACS-006).
3. AI Orchestration повторно вызывает Furniture Intelligence **не заново с нуля**, а с дополнительным ограничением `sizeConstraint`, полученным из данных о нарушении (например, "предмет `causingItemId` должен быть уже на N см") — новый параметр, добавленный к контракту `recommendFurniture` (см. обновление ACS-005).
4. Layout & Ergonomics Intelligence вызывается повторно с обновлённым списком мебели.
5. Цикл повторяется не более **ограниченного числа раз** — конкретное число не зафиксировано как жёсткая константа в этом документе (аналогично нерешённому числу циклов Refinement в ACS-010), но обязано быть настраиваемым параметром, отдельным от счётчика циклов Quality Intelligence (Refinement Engine) — это разные слои: один работает до генерации изображения (здесь), другой — после (ACS-010).
6. **Если лимит исчерпан без полного устранения нарушения:** AI Orchestration обязана передать лучший из достигнутых вариантов дальше в Prompt Intelligence, вместе с оставшимися `ergonomicIssue` как частью `domainDecisions` — не отбрасывать нарушение молча. Это сохраняет прослеживаемость для Design Reasoning (PCS-008): если компромисс по эргономике был вынужденным, пользователь (через объяснение) может об этом узнать, а не получить результат, который выглядит корректным, но таковым не является.

**Почему это решение, а не временная оценка:** аналогично решению о параллелизме между помещениями — это архитектурный выбор с явным обоснованием, а не эмпирический факт, поэтому пересматривается только через Decision Governance, а не по мере поступления данных.

## Inputs

| Поле | Обязательность | Описание |
|---|---|---|
| `structuredScenes` | Обязательно | Один или несколько Structured Scene (от Scene Intelligence, ACS-002) — по одному на помещение проекта. |
| `projectContext` | Опционально | Существующий Project Design Context, если проект уже создавался ранее (для добавления нового помещения в существующий проект). |
| `userPreferences` | Опционально | Данные от Personal AI Designer (PCS-009), если пользователь не новый. |

## Outputs

| Поле | Описание |
|---|---|
| `projectDesignContext` | Единый контекст (стиль, палитра, материалы), применяемый ко всем помещениям проекта. |
| `executionPlan` | Список вызовов Domain Intelligence и Generation Intelligence с указанием, какие из них могут выполняться параллельно, а какие — нет (согласно решению выше — генерация помещений параллельна, разрешение Project Design Context предшествует ей последовательно). |

## Dependencies

- **Восходящие:** Scene Intelligence (ACS-002) — обязательный источник Structured Scene для каждого помещения.
- **Нисходящие (потребители результата оркестрации):** Generation Intelligence (ACS-001) — получает команды на генерацию согласно `executionPlan`; все Domain Intelligence модули — вызываются в порядке, определённом Agent Framework.

**Запрещённые зависимости:** AI Orchestration не имеет права напрямую вызывать Provider Layer, минуя Generation Intelligence — весь доступ к генерации должен идти через зафиксированный контракт ACS-001, а не в обход него.

## Consumers

PCS-003 (Project Intelligence) — основной потребитель решения о параллельной обработке; PCS-004 (Commercial Design) — то же для многозоновых коммерческих проектов; PCS-001/002 — используют Agent Framework для координации Domain Intelligence даже в рамках одного помещения (не только многопроектных сценариев).

## Public Contract

```
composeProjectContext(
  structuredScenes: StructuredScene[],
  userPreferences?: UserPreferences
) => ProjectDesignContext

planExecution(
  projectContext: ProjectDesignContext,
  structuredScenes: StructuredScene[]
) => ExecutionPlan {
  parallelizableSteps: GenerationTask[],  // генерация помещений (решение о параллелизме между помещениями)
  domainIntelligenceOrder: {
    sequential: ["StyleIntelligence"],
    parallelAfterStyle: ["FurnitureIntelligence", "MaterialIntelligence", "ColorIntelligence"],
    sequentialAfterFurniture: ["LayoutErgonomicsIntelligence"]
  },
  sequentialSteps: Task[]                 // разрешение контекста стиля, предшествующее генерации
}

resolveFurnitureLayoutConflict(
  ergonomicIssues: ErgonomicIssue[],  // от Layout & Ergonomics Intelligence, ACS-006, с полем causingItemId
  maxAttempts: number                 // настраиваемый параметр, не жёстко зафиксированное число
) => {
  finalPlacementPlan: PlacementItem[],
  unresolvedIssues: ErgonomicIssue[]  // передаются дальше в Prompt Intelligence, если лимит исчерпан
}
```

## Extension Points

- Настраиваемая степень параллелизма (см. п.3 архитектурного решения) — должна быть вынесена в конфигурацию, а не зашита в код, чтобы её можно было адаптировать по мере изменения тарифных лимитов платформы.
- Пересмотр модели на последовательную с передачей визуального результата — возможен только через отдельный ADR и Decision Governance (см. п.4 архитектурного решения), не как тихое изменение поведения.
- `maxAttempts` для `resolveFurnitureLayoutConflict` — настраиваемый параметр, конкретное значение не зафиксировано в этом документе, требует решения при реализации Gate 4. Должен оставаться отдельным от максимального числа циклов Refinement в Quality Intelligence (ACS-010) — это разные слои итерации (до и после генерации изображения соответственно), и их не следует объединять в один общий счётчик.

## Knowledge Packs

Не использует Knowledge Packs напрямую — оперирует контекстом, уже собранным Domain Intelligence модулями, которые сами обращаются к Knowledge Packs.

## Tests

- **Контрактные тесты:** `planExecution` корректно относит генерацию помещений к параллелизуемым шагам, а разрешение Project Design Context — к последовательным, предшествующим им; `domainIntelligenceOrder` всегда ставит Style Intelligence первым, а Layout & Ergonomics Intelligence — после Furniture Intelligence.
- **Boundary-тесты:** AI Orchestration не вызывает Provider Layer напрямую, минуя Generation Intelligence.
- **Тесты согласованности:** при параллельной генерации нескольких помещений с одним Project Design Context результат использует один и тот же стиль/палитру (проверяется на уровне промпта, переданного в каждый вызов, а не визуально).
- **Нагрузочные тесты:** проверка поведения при разной степени параллелизма с учётом реальных rate-limit платформы на момент реализации (см. ограничение в п.3 архитектурного решения).
- **Тесты цикла разрешения конфликта:** заданный тестовый сценарий с намеренно несовместимой мебелью корректно разрешается за N попыток или явно возвращает `unresolvedIssues`, если лимит `maxAttempts` исчерпан — цикл не продолжается бесконечно и не завершается молчаливой потерей нарушения.

## Performance Expectations

Прямых собственных AI-вызовов модуль не совершает (не считая, возможно, LLM-вызова для планирования — не подтверждено, требует уточнения при реализации). Совокупный Performance Budget проекта из нескольких помещений теперь может быть оценён, в отличие от PCS-003:

При параллельной обработке (принятое решение) общее время генерации проекта из N помещений ограничено временем одного вызова (~60 сек, по данным ACS-001) плюс временем на разрешение Project Design Context — **не измерено**, и временем, необходимым на соблюдение rate-limit при параллельных запросах — **не измерено, зависит от фактического состояния тарифов платформы на момент реализации**.

Совокупная стоимость — сумма стоимости по каждому помещению (N × ≤ $0.061), не зависит от параллелизма (параллелизм влияет на время, не на суммарную стоимость вызовов).

| Параметр | Статус | Значение |
|---|---|---|
| Совокупная стоимость проекта из N помещений | Подтверждено (арифметически, на основе ACS-001) | N × ≤ $0.061 |
| Совокупное время проекта из N помещений | Оценка на основе принятого архитектурного решения | ≈ ~60 сек + время разрешения контекста (не измерено) + возможные задержки rate-limit (не измерено) |
