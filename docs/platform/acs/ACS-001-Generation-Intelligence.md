# ACS-001 — Generation Intelligence

**Статус:** AI Capability (сервисный модуль)
**Соответствует:** Living Strategic Roadmap v1.3, Level 3 — Generation Intelligence
**Обслуживает:** PCS-001 (Smart Interior Generation), PCS-002 (Intelligent Room Transformation)
**Версия документа:** 1.0

---

## Purpose

Единая точка вызова генерации изображения у провайдера. Модуль сам не решает, *что* генерировать — он получает готовый Prompt String и параметры режима, вызывает провайдера и возвращает результат. Существует именно для того, чтобы PCS-001 и PCS-002 не реализовывали каждый свою собственную интеграцию с провайдером, а использовали один общий, предсказуемый контракт.

## Responsibilities

- Принимать Prompt String (от Prompt Intelligence) и параметры генерации.
- Определять, какой режим вызова провайдера использовать, **на основании явно переданного параметра `mode`**, а не самостоятельного анализа контекста.
- Вызывать Provider Layer с корректными параметрами для выбранного режима.
- Возвращать результат: сгенерированное изображение (или ссылку на него), фактическую стоимость вызова, фактическую задержку, статус успеха/ошибки, метаданные провайдера (модель, id запроса).
- Обеспечивать единообразную обработку ошибок провайдера (таймаут, отказ, некорректный ответ) независимо от режима.

## Non-responsibilities

Модуль **не отвечает** за:
- понимание содержимого помещения (это зона ответственности Scene Intelligence);
- формирование самого текста промпта (Prompt Intelligence);
- определение маски/области для частичного редактирования (Scene Intelligence передаёт готовый регион, Generation Intelligence его не вычисляет);
- проверку качества результата — соответствует ли он геометрии, стилю, ограничениям (Quality Intelligence);
- решение о повторной генерации при неудачном результате (это решает Quality Intelligence / оркестратор, Generation Intelligence только исполняет вызов, который его попросили сделать);
- бизнес-логику (тарифы, лимиты пользователя) — это уровень Production Platform.

## Разграничение PCS-001 и PCS-002 (техническая граница)

Это ключевая фиксация, ради которой создаётся документ.

| | PCS-001 (Smart Interior Generation) | PCS-002 (Intelligent Room Transformation) |
|---|---|---|
| **Режим (`mode`)** | `FULL_GENERATION` | `PARTIAL_EDIT` или `INPAINTING` |
| **Обязательное наличие исходного изображения** | Да (референс комнаты), но используется как основа для полной перерисовки | Да, используется как основа с сохранением незатронутых зон |
| **Обязательное наличие маски/региона** | Нет | Да — без маски вызов в этом режиме не допускается |
| **Ожидаемый результат** | Полностью перерисованное изображение помещения | Изображение с изменённой только заданной зоной, остальное — визуально идентично исходному |

**Правило контракта:** Generation Intelligence обязан отклонять вызов с ошибкой, если:
- `mode = FULL_GENERATION`, но передана маска/регион (несогласованный вызов — вероятно, ошибка на стороне вызывающего PCS);
- `mode = PARTIAL_EDIT` или `INPAINTING`, но маска/регион отсутствует.

Это не бизнес-правило, а инвариант контракта — он должен проверяться тестами (см. раздел Tests), а не оставаться на усмотрение вызывающего кода.

## Inputs

| Поле | Обязательность | Описание |
|---|---|---|
| `promptString` | Обязательно | Результат работы Prompt Intelligence (Formatter). |
| `mode` | Обязательно | `FULL_GENERATION` \| `PARTIAL_EDIT` \| `INPAINTING`. |
| `sourceImage` | Обязательно для всех режимов | Референсное изображение помещения. |
| `maskRegion` | Обязательно для `PARTIAL_EDIT`/`INPAINTING`, запрещено для `FULL_GENERATION` | Область изображения, подлежащая изменению. |
| `size` | Обязательно | Например, 1024×768 или 1024×1024 (см. подтверждённые размеры в PCS-001). |
| `qualityTier` | Обязательно | Например, "низкое"/"среднее" — согласно тарифной таблице провайдера. |

## Outputs

| Поле | Описание |
|---|---|
| `resultImage` | Сгенерированное изображение (URL или бинарные данные). |
| `status` | `success` \| `error`, с кодом ошибки при неудаче. |
| `actualCost` | Фактическая стоимость вызова (для сверки с Performance Budget). |
| `actualLatency` | Фактическая задержка вызова. |
| `providerMetadata` | Название модели, id запроса у провайдера — нужно для трассируемости и Decision Governance (Evidence Collection). |

## Dependencies

- **Prompt Intelligence** (восходящая) — единственный источник `promptString`. Generation Intelligence не формирует и не модифицирует промпт самостоятельно.
- **Scene Intelligence** (восходящая, только для режимов редактирования) — источник `maskRegion`.
- **Provider Layer** (используется, не является отдельным AI Capability верхнего уровня, а внутренним слоем этого модуля) — фактическая интеграция с `gpt-image-2/edit` или другим провайдером.

**Запрещённые зависимости** (согласно правилу ACS из Roadmap v1.3 — список запрещённых зависимостей обязателен для каждого модуля):
- Generation Intelligence не имеет права напрямую обращаться к Domain Intelligence, Knowledge Packs или Production-коду — все необходимые данные должны прийти через `promptString` и явные параметры вызова.
- Не имеет права принимать решения о повторной генерации — это вызвало бы скрытую зависимость от логики Quality Intelligence внутри модуля, которому это не принадлежит по Responsibilities.

## Consumers

- PCS-001 (Smart Interior Generation) — вызывает с `mode = FULL_GENERATION`.
- PCS-002 (Intelligent Room Transformation) — вызывает с `mode = PARTIAL_EDIT` или `INPAINTING`.
- Потенциально Quality Intelligence (Refinement Engine, Phase K) — при необходимости повторного вызова после неудачной валидации; повторный вызов инициируется извне, сам модуль не запускает себя рекурсивно.

## Public Contract

Функциональный контракт (без привязки к конкретному языку/фреймворку — техническая реализация фиксируется отдельно в коде):

```
generate(params: {
  promptString: string,
  mode: "FULL_GENERATION" | "PARTIAL_EDIT" | "INPAINTING",
  sourceImage: ImageRef,
  maskRegion?: RegionDescriptor,   // обязателен только для PARTIAL_EDIT / INPAINTING
  size: Dimensions,
  qualityTier: "low" | "medium"
}) => {
  status: "success" | "error",
  resultImage?: ImageRef,
  actualCost: number,
  actualLatency: number,
  providerMetadata: ProviderMetadata,
  errorCode?: string
}
```

## Extension Points

- Добавление нового провайдера (помимо `gpt-image-2/edit`) не должно менять публичный контракт — расширяется только внутренняя реализация Provider Layer.
- Добавление нового режима (например, `MULTI_STAGE` для Phase H — Generation Intelligence в широком смысле Roadmap) требует явного расширения enum `mode` и отдельного ADR, а не неявного расширения поведения существующих режимов.

## Knowledge Packs

Не использует Knowledge Packs напрямую — все знания о стиле/материалах уже "свёрнуты" в `promptString` до вызова этого модуля. Это намеренное ограничение, обеспечивающее простоту тестирования и независимость модуля от роста Domain Intelligence.

## Tests

Согласно Architecture Boundary Tests (Phase C1) и правилам ACS:

- **Контрактные тесты:** вызов с `mode = FULL_GENERATION` и переданной маской должен завершаться ошибкой; вызов с `mode = PARTIAL_EDIT`/`INPAINTING` без маски — тоже ошибкой.
- **Boundary-тесты:** Generation Intelligence не импортирует Domain Intelligence, Knowledge Packs, Production-код напрямую.
- **Golden-тесты:** фиксированный набор `promptString` + `mode` даёт стабильно корректный (по формату, не по художественному качеству) результат вызова провайдера.
- **Тесты обработки ошибок:** таймаут и отказ провайдера обрабатываются единообразно независимо от режима.

## Performance Expectations

Раздельно по режимам — не единый бюджет, так как подтверждённые данные есть только для одного из них.

| Режим | Стоимость | Задержка | Статус данных |
|---|---|---|---|
| `FULL_GENERATION` | ≤ $0.061 (среднее качество, 1024×1024) | ~60 сек | Подтверждено (см. PCS-001) |
| `PARTIAL_EDIT` / `INPAINTING` | ≤ $0.061 (предположение по аналогии) | ~60 сек (предположение по аналогии) | Не подтверждено — требует отдельного замера (см. PCS-002) |

До получения отдельных подтверждённых данных для режимов редактирования Performance Budget для них не может считаться зафиксированным окончательно — только рабочей оценкой.

## ADR References

Не заполнено — требует создания отдельного ADR, фиксирующего именно это разграничение режимов (`mode`) как архитектурное решение, а не только описание в этом документе. Рекомендуется оформить как отдельный ADR перед началом реализации Gate 1/Gate 5.
