# ACS-007 — Material Intelligence

**Статус:** AI Capability (Domain Intelligence, сервисный модуль)
**Соответствует:** Living Strategic Roadmap v1.3, Level 3 — Domain Intelligence → Material Intelligence
**Обслуживает:** Prompt Intelligence (ACS-004) как поставщик решений; PCS-005 (Product Intelligence) и PCS-006 (Renovation Intelligence) как потребители данных о материалах
**Версия документа:** 1.0

---

## Purpose

Знания о материалах, текстурах, сочетаниях, отражении света — согласно формулировке Domain Intelligence в более ранних версиях Roadmap. Развивается по продуктовой потребности: явно указана как модуль, используемый Renovation Intelligence (PCS-006) — расчёт количества материалов по площади невозможен без параметров самого материала (например, размер плитки, ширина рулона обоев).

## Responsibilities

- Классифицировать материал, обнаруженный Scene Intelligence на изображении (тип, приблизительная фактура) — для сценариев замены материала (PCS-002, локальное изменение стен/пола/потолка) и для сопоставления с товарами (PCS-005, Material Matching).
- На основании выбранного стиля (решение Style Intelligence, получаемое через AI Orchestration) — предлагать подходящие материалы для стен, пола, потолка, плитки.
- Проверять сочетаемость материалов между собой (например, тип напольного покрытия и тип плитки в одном помещении) и фиксировать несовместимые сочетания явно.
- Предоставлять параметры материала, необходимые для расчёта количества (PCS-006): например, размер единицы плитки, ширина и длина рулона обоев, площадь покрытия на упаковку. **Это обязательное требование**, зафиксированное как зависимость в PCS-006 — без этих данных Renovation Intelligence не может функционировать, а не опциональное расширение.
- Каждое решение сопровождать `sourceRule` (или явным `null`), аналогично требованию из ACS-004, ACS-005, ACS-006.

## Non-responsibilities

Модуль **не отвечает** за:
- выбор цвета как такового — это зона ответственности Color Intelligence; однако между материалом и цветом есть содержательное пересечение (например, "дуб" одновременно является материалом и подразумевает определённый диапазон цвета) — **точная граница между Material Intelligence и Color Intelligence не определена в этом документе** и требует согласования при составлении ACS для Color Intelligence, а не решается здесь в одностороннем порядке;
- тип и размер мебели — Furniture Intelligence (ACS-005);
- расположение материала в пространстве — не применимо в том смысле, что материалы покрывают всю поверхность (стену, пол), а не позиционируются как отдельные объекты, поэтому Layout & Ergonomics Intelligence к этому модулю прямого отношения не имеет;
- итоговую генерацию изображения — Generation Intelligence (ACS-001);
- итоговый расчёт бюджета — Renovation Intelligence (PCS-006) потребляет данные отсюда, но сам расчёт делает самостоятельно.

## Известный технический контекст, требующий решения при реализации

Это зафиксировано честно, а не домыслено или скрыто.

По данным памяти о предыдущих сессиях работы над проектом, ранее использовавшаяся модель (`flux-pro/v1/canny`) не интерпретировала HEX-коды цвета напрямую — для конвертации HEX в описательные названия был построен отдельный `hexToColorDescription` helper (~80 значений цветовой карты + RGB-фолбэк с определением бежевого/терракотового). Кроме того, был отдельно зафиксирован workaround: добавление фразы "white ceiling, ceiling color is white not colored" ко всем промптам в режиме смены стиля — чтобы модель не перекрашивала потолок в цвет стен.

**Решение принято в ACS-008 (Color Intelligence):** оба механизма (конвертация HEX → описательное название, явное решение о цвете потолка) закреплены за Color Intelligence, а не за этим модулем. Актуальность обеих проблем для текущей модели провайдера по-прежнему не подтверждена повторно — см. ACS-008 для деталей и плана проверки.

## Inputs

| Поле | Обязательность | Описание |
|---|---|---|
| `sceneObject` | Обязательно для классификации существующего материала | Дескриптор поверхности от Scene Intelligence (ACS-002). |
| `spaceType` | Обязательно для рекомендаций с нуля | Тип пространства. |
| `styleDecision` | Обязательно | Решение Style Intelligence, полученное через AI Orchestration. |

## Outputs

| Поле | Описание |
|---|---|
| `materialClassification` | Для существующей поверхности: тип материала, приблизительная фактура, `sourceRule`. |
| `materialRecommendation[]` | Для генерации с нуля или замены: список подходящих материалов с `sourceRule`. |
| `materialSpec` | Параметры для расчёта количества (размер единицы, площадь покрытия) — обязательное поле для потребления PCS-006. |
| `compatibilityIssue[]` | Несовместимые сочетания материалов, если есть. |

## Dependencies

- **Восходящие:** Scene Intelligence (ACS-002); AI Orchestration (ACS-003) — единственный источник `styleDecision`.
- **Нисходящие (потребители):** Prompt Intelligence (ACS-004); Product Intelligence (PCS-005, Material Matching); Renovation Intelligence (PCS-006) — прямой потребитель `materialSpec`, обязательная зависимость, зафиксированная ранее.

**Запрещённые зависимости:** те же ограничения, что в ACS-005 и ACS-006 — не вызывает Provider Layer, Prompt Intelligence, другие Domain Intelligence модули напрямую, включая Color Intelligence (несмотря на содержательное пересечение, отмеченное выше — разрешение конфликта между их решениями происходит на уровне Rule Engine в Prompt Intelligence, ACS-004, а не прямой договорённостью между модулями).

## Consumers

Prompt Intelligence (ACS-004), Product Intelligence (PCS-005), Renovation Intelligence (PCS-006).

## Public Contract

```
classifyMaterial(sceneObject: SceneObjectDescriptor) => {
  type: string,
  texture: string,
  sourceRule: string | null
}

recommendMaterial(
  spaceType: string,
  surfaceType: "wall" | "floor" | "ceiling" | "tile",
  styleDecision: StyleDecision
) => MaterialRecommendation[]  // { type, materialSpec, sourceRule }

checkMaterialCompatibility(materials: MaterialItem[]) => CompatibilityIssue[]
```

## Extension Points

- Расширение на новые типы поверхностей и материалов по мере подключения новых Knowledge Packs.
- Разрешено в ACS-008: `hexToColorDescription` и правило про белый потолок закреплены за Color Intelligence, не за этим модулем.

## Knowledge Packs

**Residential Materials** и **Commercial Materials** — по аналогии с принципом Commercial Dimension, применённым в ACS-005 и ACS-006 (точное имя пакетов не зафиксировано в Roadmap v1.3 напрямую для Material Intelligence — использую по аналогии, требует подтверждения при реализации).

## Tests

- **Контрактные тесты:** каждое решение сопровождается `sourceRule` или явным `null`.
- **Boundary-тесты:** модуль не вызывает Color Intelligence, Furniture Intelligence, Prompt Intelligence, Provider Layer напрямую.
- **Тесты `materialSpec`:** параметры, необходимые для расчёта количества (PCS-006), присутствуют для каждого рекомендованного материала — их отсутствие должно считаться ошибкой контракта, а не допустимым пропуском.
- **Регрессионный тест на HEX/потолок** (при подтверждении актуальности для текущей модели, см. открытый вопрос выше) — если решение принято в пользу этого модуля, а не Color Intelligence.

## Performance Expectations

**Не определено.** Тот же класс открытого вопроса, что в ACS-004, ACS-005, ACS-006 — детерминированная логика против LLM-вызова для классификации материала на неоднозначных изображениях.

## ADR References

Рекомендуется отдельный ADR для разрешения границы Material/Color Intelligence в части HEX-конвертации и правила про потолок — этот вопрос уже возникает третий раз в связанных модулях (Formatter в ACS-004, теперь здесь) и не должен решаться неявно по мере реализации каждого модуля отдельно.
