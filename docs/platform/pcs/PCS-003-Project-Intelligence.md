# PCS-003 — Project Intelligence

**Статус:** Platform Capability
**Соответствует:** Living Strategic Roadmap v1.3, Level 2, Capability 3
**Версия документа:** 1.0

---

## Purpose

Ввести проект как основную сущность платформы вместо разовой генерации одной комнаты. Проект объединяет несколько помещений — комнату, квартиру, дом, офис, ресторан, магазин, гостиницу, клинику — под единым стилем, палитрой и набором правил.

Согласно Roadmap v1.3, Apartment Projects и Commercial Projects (упомянутые как отдельные пункты в более ранних версиях Roadmap) объединяются в эту единую архитектуру, а не реализуются как две параллельные системы.

## Business Value

- Меняет модель использования продукта с "одна генерация — один результат" на "проект — множество согласованных результатов", что увеличивает вовлечённость и время в продукте.
- Прямая предпосылка для Renovation Intelligence (PCS-006, ожидается): расчёт бюджета ремонта имеет смысл только на уровне всего проекта (квартиры/объекта), а не отдельной комнаты.
- Для коммерческого сегмента (Capability 4) — единственный реалистичный способ продавать продукт бизнесу: ни один ресторан или отель не заказывает дизайн одной случайной комнаты в отрыве от остальных помещений.

## User Stories

- Как пользователь, я хочу создать проект "моя квартира" и последовательно сгенерировать прихожую, кухню, гостиную, спальню и детскую в едином стиле.
- Как пользователь, я хочу, чтобы при изменении стиля в одной комнате проекта система предложила согласованные изменения для остальных помещений (не автоматически применяла, а именно предлагала).
- Как владелец кафе, я хочу спроектировать зал, бар и кухню как один проект с общей цветовой палитрой и материалами.
- Как пользователь, я хочу вернуться к своему проекту позже и продолжить работу с того места, где остановился, не начиная заново.

## AI Modules

- **AI Orchestration** (AI Orchestrator, Context Composer) — координирует последовательность генераций внутри одного проекта, обеспечивая, что контекст (стиль, палитра) передаётся от одной комнаты к другой.
- **Generation Intelligence** (см. ACS-001) — вызывается многократно, по одному разу на помещение, всегда в режиме `FULL_GENERATION` (если проект создаётся с нуля) или `PARTIAL_EDIT`/`INPAINTING` (если пользователь позже редактирует отдельное помещение внутри уже существующего проекта). Project Intelligence не вводит новый режим генерации — переиспользует существующий контракт ACS-001.
- **Style Intelligence, Material Intelligence, Color Intelligence** — обеспечивают согласованность между помещениями (Cross-room Style Consistency).
- **Scene Intelligence** — применяется отдельно к каждому помещению проекта, результаты агрегируются на уровне проекта.

## Knowledge Packs

Использует те же пакеты, что и PCS-001/PCS-002 (Residential, Commercial и далее), но добавляет требование: при работе в рамках одного проекта все помещения должны использовать один и тот же активный Knowledge Pack (нельзя одновременно применять Residential-логику к кухне и Hospitality-логику к спальне внутри одной квартиры) — за исключением смешанных объектов, где это архитектурно оправдано (например, гостиница с жилой частью для персонала); эта ситуация требует отдельного правила и не покрывается в первой реализации.

## Dependencies

- **Восходящие:** PCS-001 (Smart Interior Generation) и PCS-002 (Intelligent Room Transformation) — Project Intelligence не заменяет их, а оркестрирует множественные вызовы к ним через уже зафиксированный контракт Generation Intelligence (ACS-001).
- **Нисходящие:** Product Intelligence (PCS-005, ожидается) и Renovation Intelligence (PCS-006, ожидается) оперируют на уровне проекта, а не отдельной комнаты — то есть зависят от того, что Project Intelligence уже реализована.
- **Внешние:** отсутствуют сверх тех, что уже зафиксированы для PCS-001/002 (Provider Layer).

## Integration Gates

Согласно Engineering Roadmap (Level 4): **Gate 6 (Phase H — Project Intelligence)**.

Предварительные условия прохождения (требуют уточнения при планировании Phase H):
- проект из минимум 3 помещений сгенерирован с согласованным стилем без ручной корректировки между комнатами;
- пользователь может вернуться к ранее созданному проекту и добавить новое помещение с сохранением уже заданного стиля;
- изменение одного помещения не разрушает данные остальных помещений проекта.

## Performance Budget

Отдельных подтверждённых данных для этой capability в рамках диалога не предоставлялось — она не выполняет собственных AI-вызовов, а оркестрирует уже описанные в PCS-001/002/ACS-001. Совокупная стоимость и время проекта — это сумма стоимости и времени по каждому помещению (см. подтверждённые данные ACS-001 для `FULL_GENERATION`: ≤ $0.061 и ~60 сек за помещение), плюс накладные расходы на оркестрацию (координация контекста между вызовами), которые пока не измерены.

| Параметр | Статус | Значение |
|---|---|---|
| Стоимость на одно помещение проекта | Подтверждено (унаследовано из ACS-001) | ≤ $0.061 |
| Время на одно помещение проекта | Подтверждено (унаследовано из ACS-001) | ~60 сек |
| Обработка нескольких помещений | Решение принято в ACS-003 (AI Orchestration) | Параллельная генерация, с предварительным последовательным разрешением Project Design Context |
| Накладные расходы на оркестрацию | Не измерено | Зависит от времени разрешения контекста и фактических rate-limit платформы — см. ACS-003 |
| Суммарное время/стоимость для проекта из нескольких помещений | Оценка на основе принятого решения | Стоимость: N × ≤ $0.061 (подтверждено арифметически). Время: ≈ ~60 сек + неизмеренные накладные расходы — см. ACS-003 |

**Решение принято.** Вопрос параллельной/последовательной обработки помещений закрыт в ACS-003 (AI Orchestration): выбрана параллельная генерация с единым Project Design Context, формируемым до начала генерации любого помещения. Подробное обоснование и ограничения (влияние rate-limit платформы) — см. ACS-003.

## Success Criteria

Фиксируются до начала реализации Gate 6. Предварительные формулировки:

- Проект из нескольких помещений демонстрирует единый стиль и палитру без ручной постобработки пользователем.
- Повторный визит пользователя к существующему проекту не требует повторного ввода стилевых предпочтений.
- Добавление нового помещения в проект не нарушает ранее сгенерированные результаты.

## KPIs

- **Технический профиль (ранние этапы):** успешность согласованной генерации нескольких помещений подряд, стабильность передачи контекста между вызовами, накладные расходы на оркестрацию.
- **Продуктовый профиль (после появления пользователей):** среднее количество помещений на проект, доля пользователей, возвращающихся к ранее созданному проекту, завершение проекта (все запланированные помещения сгенерированы) против брошенных на середине.

## Future Evolution

- Автоматическое предложение "следующего логичного помещения" для генерации на основе типа проекта (например, после кухни предложить гостиную для квартиры) — требует отдельной оценки полезности после появления реальных пользовательских данных.
- Смешанные объекты (жилая + коммерческая часть в одном проекте, например мини-отель) — отдельная архитектурная задача, не покрыта в первой версии.
- Совместное редактирование проекта несколькими пользователями — прямое пересечение с Collaboration Platform (PCS-010, ожидается), требует координации при проектировании обеих capability.
