# VistaRoom AI — ADR Numbering Policy (Governance)

## Status

Accepted
Accepted by: Project Owner
Date: 2026-07-08
Revision: 2

## Purpose

Настоящий документ устанавливает правило резервирования и присвоения номеров ADR 
в проекте VistaRoom AI. Это процессная конвенция, а не архитектурное или 
инженерное решение — она регулирует, как именно номер ADR становится закреплённым 
за темой, а не что должно быть решено по существу какой-либо темы.

## Scope

Настоящая политика применяется ко всем новым ADR, создаваемым после её принятия. 
Не требует ретроактивного изменения уже существующих ADR или исторических записей 
ADR Index, кроме случаев исправления очевидных ошибок.

## 1. Резервируют ли записи в backlog номера ADR?

Нет. Упоминание номера ADR в `ADR-Backlog-Consolidated.md` (или аналогичном 
backlog-документе) является предварительным, непринятым указанием возможного 
будущего номера, а не резервированием. Backlog-запись не создаёт обязательства 
присвоить именно этот номер именно этой теме — она носит справочный характер до 
момента фактического создания ADR.

## 2. Что является единственным источником истины для нумерации?

Единственным источником истины является `ADR_INDEX.md` (ADR Registry). Номер 
считается занятым тогда и только тогда, когда в `ADR_INDEX.md` присутствует 
запись с этим номером, соответствующая реально существующему файлу ADR в 
репозитории. Любые упоминания номеров ADR в backlog, черновиках, чатах или иных 
документах не являются источником истины и не заменяют проверку по `ADR_INDEX.md`.

## 3. Когда именно присваивается номер ADR?

Номер считается официально присвоенным тогда, когда файл ADR и соответствующая 
запись в `ADR_INDEX.md` входят в один логически завершённый набор изменений 
(change set), обычно оформляемый одним commit. Это описывает конечное состояние, 
а не внутреннюю последовательность локальных действий (создание файла, тесты, 
обновление индекса и т.д.) — до фиксации change set номер не считается 
присвоенным, независимо от порядка промежуточных шагов.

## 4. Как разрешаются коллизии между backlog и фактически создаваемыми ADR?

Если тема, готовящаяся к оформлению как новый ADR, конфликтует по номеру с 
существующей backlog-пометкой:

- Приоритет имеет тема, которая фактически создаётся первой согласно этому 
  правилу (п. 3) — то есть та, чей change set (файл + запись в `ADR_INDEX.md`) 
  фиксируется раньше.
- Тема, backlog-номер которой оказался занят, получает следующий свободный номер 
  на момент её фактического создания.
- После присвоения нового номера синхронизация backlog должна быть выполнена при 
  первом последующем редактировании соответствующей backlog-записи.
- Коллизия не разрешается "задним числом" переприсвоением уже созданных и 
  принятых ADR — правило действует только на будущие, ещё не созданные документы.

## 5. Рекомендуемый порядок действий при создании нового ADR

1. Подготовить draft документа без окончательного номера (допускается 
   использование временного имени или "TBD"). До присвоения номера документ 
   остаётся draft, а не ADR в статусе Proposed.
2. Выполнить сверку по `ADR_INDEX.md`: убедиться, какой номер является следующим 
   свободным на момент создания.
3. Проверить `ADR-Backlog-Consolidated.md` на предмет существующей пометки для 
   этого номера — если тема backlog-пометки отличается от темы готовящегося 
   документа, применить правило разрешения коллизии (п. 4).
4. Присвоить номер, создать физический файл ADR и добавить запись в 
   `ADR_INDEX.md` одним change set. С этого момента документ становится ADR в 
   статусе Proposed.
5. Только после этого проводить цикл Proposed → Revision Requested → Revision N 
   → Accepted согласно `ADR-Authoring-Convention.md`.

## Non-goals

- Документ не решает вопросы содержания какого-либо конкретного ADR, включая 
  ADR-006 Integration.
- Не создаёт новых архитектурных решений.
- Не относится к Implementation Package.
- Не изменяет формат Status или Lifecycle, уже зафиксированные в 
  `ADR-Authoring-Convention.md`.
