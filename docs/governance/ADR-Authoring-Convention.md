VistaRoom AI — ADR Authoring Convention (Governance Style Guide)
Рекомендуемое расположение: docs/governance/ADR-Authoring-Convention.md
Purpose
Настоящий документ описывает правила оформления Architecture Decision Records (ADR) в проекте VistaRoom AI. Это не архитектурное и не инженерное решение — это конвенция ведения документации, дополняющая, а не заменяющая governance-принципы, зафиксированные в Project Context.
Scope
Применяется ко всем новым ADR, создаваемым после принятия настоящей конвенции. Не требует и не предполагает ретроактивного изменения уже принятых ADR (ADR-000–ADR-007) — расхождение формата у исторических документов признаётся допустимым.
Status Block Format
Для документов в статусе Proposed / Revision Requested
## Status

Proposed
Revision: N
Для документов в статусе Accepted
## Status

Accepted
Accepted by: Project Owner
Date: YYYY-MM-DD
Revision: N
Примечания:

Поле даты называется Date (не Acceptance date) — выбрано для совпадения с уже существующим прецедентом ADR-007, минимизируя расхождение стилей.
Поле Revision фиксирует итоговый номер ревизии на момент принятия, отражая фактический цикл Proposed → Revision Requested → Revision N → Accepted.
Промежуточные ревизии не описываются словами в самом поле Status (например, недопустимо: "Revision 3 (final, wording clarification applied...)") — история ревизий и правок остаётся в git-истории документа, а не в тексте Status.

ADR Lifecycle
Новый ADR проходит следующие стадии:
Proposed
→ Revision Requested (при необходимости)
→ Revision N
→ Accepted
После получения статуса Accepted содержательное изменение ADR не допускается.
Допускаются только редакционные изменения, не изменяющие смысл документа (например, исправление опечаток, форматирования, внутренних ссылок или терминологии). Такие изменения не требуют новой ревизии и не изменяют статус документа.
Related Documents Section
Каждый новый ADR, если он логически связан с другими ADR/ACS/ED, обязан содержать раздел Related Documents со списком ссылок и кратким указанием характера связи (например, "фиксирует контракт, чьи границы определяет..."). Связь с другими документами отражается через ссылки в этом разделе, а не через включение названия связанного документа в заголовок ADR (см. прецедент переименования "ADR-006 Integration Decision" → "Generation Intelligence Architectural Boundaries" в этой сессии).
Naming Convention
Заголовок ADR должен описывать предмет решения, а не документ, с которым он связан. Например: "Generation Intelligence Architectural Boundaries", а не "ADR-006 Integration Decision" — связь с ADR-006 указывается в Related Documents.
Non-goals

Этот документ не устанавливает формат ED — он может быть предметом отдельного решения при необходимости.
Не переоткрывает содержание ADR-005, ADR-007, ED-004 и не требует их правки.
Не определяет правила резервирования номеров ADR. Этот вопрос регулируется отдельно.
