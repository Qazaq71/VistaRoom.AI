## 3. Per-Candidate Analysis

### C1 — Resolution Phase (ADR-009 Decision 2)
- **Архитектурная связность:** высокая с уже принятым ADR-009 (Production Integration Contract). Решение архитектурно уже оформлено — требуется инженерная реализация, а не новое ADR.
- **Зависимости:** прямое продолжение завершённого ADR-006 Implementation Package (Dimensions boundary preparation зафиксирована как изолированный артефакт, ожидающий интеграции).
- **Влияние на Platform Vision:** ограниченное — техническая доработка контракта генерации, не вводит новую Platform Capability.
- **Риск:** низкий. Решение существует, требуется Implementation Package по уже принятой архитектуре.

### C2 — Prompt Engine (full lifecycle)
- **Архитектурная связность:** высокая с Prompt Philosophy (Project Context v2.1/2.2): Prompt Pipeline → Formatter (ADR-005, реализован) → Prompt Engine (не архитектурирован). Это следующее звено в уже определённой цепочке.
- **Зависимости:** формально зависит от завершённых ADR-005/ADR-006. Возможная содержательная зависимость от C3 (ADR-004 gap review) — если `RoomContext`/`SpaceType` являются частью контекста, который Prompt Engine должен обрабатывать при полном lifecycle orchestration. Эта зависимость **не подтверждена** и требует проверки, а не предполагается как факт.
- **Влияние на Platform Vision:** высокое — Prompt Engine отвечает за оркестрацию полного жизненного цикла промпта, что напрямую влияет на Capability First принцип (промпт как основной интерфейс между пользовательским замыслом и AI-генерацией).
- **Риск:** высокий с точки зрения governance — `refinePromptDraft` и полный lifecycle **не оформлены ни одним ADR**. Реализация этого кандидата с высокой вероятностью потребует нового ADR до начала Implementation Package.

### C3 — ADR-004 gap review
- **Архитектурная связность:** прямая с ADR-005/ACS-004 (терминологическая несовместимость уже зафиксирована как unconfirmed). Косвенная — с ADR-007 (`StructuredScene`/`ProjectDesignContext` null-плейсхолдер), поскольку `RoomContext`/`SpaceType` концептуально относятся к той же предметной области (представление сцены/комнаты), которую ADR-007 сознательно оставил нереализованной.
- **Зависимости:** это анализ, а не реализация — не имеет инженерных предпосылок. Может быть выполнен независимо и в первую очередь.
- **Влияние на Platform Vision:** потенциально высокое, если несовместимость окажется не только терминологической, а контрактной — тогда потребуется отдельное Architecture Decision Required, затрагивающее уже принятые ADR-005/ACS-004.
- **Риск:** сам gap review — низкий риск (это оценка, не изменение). Риск отложенного рассмотрения — высокий: если C2 (Prompt Engine) будет спроектирован до завершения C3, а gap review впоследствии выявит контрактный конфликт, это потребует пересмотра уже выполненной архитектурной работы по C2.

### C4 — `MY_STYLE_ID` migration
- **Архитектурная связность:** локальная, определена ACS-009.
- **Зависимости:** явно привязана к Gate 4, не к Gate 2/3.
- **Влияние на Platform Vision:** низкое в горизонте Gate 2.
- **Риск:** низкий при отложении; риск возрастает только если другие кандидаты (C1, C2) неявно затрагивают тот же слой идентификаторов стилей — это не подтверждено.

### C5 — Material ↔ Color conflict priority (ACS-003)
- **Архитектурная связность:** средняя-высокая с C2 (Prompt Engine) — семантическое разрешение конфликтов логически относится к той же зоне ответственности, что и `refinePromptDraft` (Rule Engine в Gate 1 ограничен детерминированной структурной валидацией).
- **Зависимости:** вероятно требует C2 как предпосылку — до появления Prompt Engine нет компонента, который бы использовал разрешённые приоритеты в рантайме.
- **Влияние на Platform Vision:** среднее — влияет на качество и предсказуемость генерации, не вводит новую Capability.
- **Риск:** низкий сам по себе, но преждевременная реализация без C2 может создать логику, которую негде использовать.

### C6 — `hexToColorDescription` relevance
- **Архитектурная связность:** узкая, локализована в Formatter (ADR-005) / ACS-004 Extension Points.
- **Зависимости:** независима от C1–C5, может быть закрыта как точечное техническое исследование.
- **Влияние на Platform Vision:** минимальное.
- **Риск:** низкий, вне зависимости от того, в каком Gate это будет рассмотрено.

### C7 — `negative prompt` gap
- **Архитектурная связность:** низкая, pre-existing, не регрессия.
- **Зависимости:** естественно всплывает при любой доработке Prompt Pipeline/Formatter (пересекается с C2).
- **Влияние на Platform Vision:** минимальное в текущем виде.
- **Риск:** низкий; не требует отдельного решения вне контекста C2.

## 8. C8 — Semantic Spatial Intelligence Core / Scene Understanding / Functional Scene Graph

**Working name:** C8 — Semantic Spatial Intelligence Core
**Источник:** Owner Direction (стратегический кандидат, добавлен по инициативе владельца проекта)

### 8.1 Что это за capability

Способность платформы строить, хранить, обновлять и запрашивать структурированную функциональную модель помещения: тип комнаты, стены, окна, двери, мебель, проходы, освещение, свободное пространство, ограничения, возможные дизайнерские действия и базовые причины дизайнерских решений.

### 8.2 Какую новую способность платформы создаёт

Переход от операций на уровне изображения/промпта к операциям на уровне структурированного представления пространства. Это позволяет платформе отвечать на вопросы о помещении и рассуждать о нём, а не только визуально его преобразовывать.

### 8.3 Отличие от Prompt Engine и generation pipeline

Prompt Engine (C2) и generation pipeline (ADR-005/006, C1, C5, C6, C7) работают на уровне подготовки и форматирования текстового промпта для image-модели — то есть оперируют представлением, потребляемым генеративной моделью. C8 вводит принципиально иной слой: структурированное, машиночитаемое представление сцены, независимое от какой-либо конкретной image-модели или промпта. Это разные уровни абстракции: C1/C2/C5/C6/C7 относятся к Generation-слою, C8 — к Representation-слою, предшествующему генерации.

### 8.4 Связь с долгосрочной целью AI Interior Designer

Долгосрочное видение проекта (согласно зафиксированному контексту) включает понимание помещения, принятие дизайнерских решений, объяснение и ведение проекта. C8 является предпосылкой для всех этих способностей: без структурированного представления пространства невозможны ни machine-проверяемые дизайнерские рассуждения, ни объяснимость решений, ни multi-room согласованность. В этом смысле C8 напрямую соответствует Capability First принципу — это capability, а не engineering improvement существующей.

### 8.5 Связь с ADR-007 (StructuredScene / ProjectDesignContext null placeholder)

Прямая. ADR-007 зафиксировал `StructuredScene`/`ProjectDesignContext` как null-плейсхолдер — сознательный архитектурный gap, поскольку продюсера этих типов в кодовой базе не существует (Room Analyzer не реализован). C8 — это ровно тот кандидат, который впервые вводит продюсера для этих типов. Реализация C8 без пересмотра ADR-007 невозможна: ADR-007 фиксировал *отсутствие* реализации как временное решение, а не запрещал её — но сам механизм построения `StructuredScene` нигде не архитектурирован.

### 8.6 Затрагиваемые PCS / ACS / ADR / ED

- **ADR-007** — потребует пересмотра или superseding: null-плейсхолдер перестаёт быть актуальным описанием, если появляется продюсер.
- **ADR-004** — терминологическая область (`RoomContext`/`SpaceType`/`SpatialPromptContext`) прямо пересекается с предметной областью C8; gap review (C3) фактически становится частью работы над C8, а не независимым от неё пунктом.
- **ACS-002, ACS-003** — согласно зафиксированному состоянию, эти документы уже существуют как capability-спецификации для `StructuredScene`/`ProjectDesignContext`, но без реализованного продюсера. C8 — это именно то, что переводит их из "документ без реализации" в активную capability.
- **ACS-004 / ADR-005** — потенциально затронуты, если структурированная сцена в будущем становится входом для Formatter; связь не установлена, требует отдельной проверки.
- **PCS** — новая Platform Capability потребует соответствующей PCS-записи (Capability First принцип: не может быть AI-модуля без предшествующей Platform Capability).

### 8.7 Требуется ли новый ADR до implementation planning

**Architecture Decision Required.**

Ни один из принятых ADR (000–009) не описывает архитектуру построения, хранения и обновления scene graph, схему сущностей/отношений, а также границы Perception-слоя (откуда берутся данные — CV-модель, ручной ввод, гибрид). ADR-007 фиксирует только *отсутствие* реализации, а не архитектуру будущей. Требуется новое ADR, предположительно по теме: "Scene Graph Schema & Producer Architecture" (включая решение о месте Room Analyzer в системе и его границах ответственности). Возможно потребуется отдельное решение по границе Perception-слоя (источник структурированных данных о помещении), если это не покрывается первым ADR.

Это не решение, принимаемое в рамках настоящего документа — оно зафиксировано здесь как предварительное governance-требование к любому implementation planning по C8.

### 8.8 Требуется ли новый или обновлённый ACS

Да, вероятно в двух формах: (1) активация/обновление уже существующих ACS-002/ACS-003 из статуса "документ без продюсера" в статус актуальной capability-спецификации; (2) возможно новый ACS для evaluation framework, измеряющего качество понимания пространства (раздел 8.13), поскольку такой framework не описан ни в одном существующем ACS.

### 8.9 Тип кандидата: архитектурный, продуктовый или engineering

Преимущественно **архитектурный** кандидат с существенным продуктовым/стратегическим измерением. Он не сводится к engineering-задаче внутри уже принятой архитектуры (в отличие от C1), и не является точечным техническим вопросом (в отличие от C6/C7). C8 определяет, какой тип платформы строится дальше — это затрагивает Platform Vision, а не только реализацию.

### 8.10 Зависимости C8

- **C3 (ADR-004 gap review)** — становится не независимым, а фактически предварительным шагом/составной частью работы над C8, поскольку терминология `RoomContext`/`SpaceType` относится к той же предметной области.
- **Perception-слой (источник данных о сцене)** — новая, ранее не оценённая зависимость: ни один принятый ADR не определяет, каким образом платформа получает структурированные данные о помещении (компьютерное зрение, ручная разметка, гибридный подход). Это открытый вопрос, а не установленный факт архитектуры.
- Зависимость от C2 (Prompt Engine) — не установлена в текущей архитектуре; возможна в будущем, если scene graph станет источником данных для промпт-конструирования, но это гипотеза, не факт.

### 8.11 Риски, если Gate 2 строится вокруг C8

- Architecture Decision Required возникает до начала любого implementation planning — Gate 2 не может начаться с реализации, только с архитектурного проектирования.
- Perception-слой не оценён архитектурно — масштаб и границы этой зависимости неизвестны до отдельного анализа.
- Возможная необходимость пересмотра ADR-007 создаёt дополнительный governance-шаг, которого не было бы при выборе Option A.
- Существующий, уже верифицированный generation pipeline (ADR-005/006) может потребовать переосмысления точки интеграции, если структурированная сцена станет частью production-контура — эта связь не установлена, но является потенциальным следствием.

### 8.12 Риски, если C8 отложить

- Платформа продолжает углублять инвестиции в generation-слой (C1/C2/C5/C6/C7), что в будущем может потребовать ретроактивной интеграции со scene understanding, если оно будет добавлено позже.
- Дифференциация платформы («AI Interior Designer» vs «инструмент визуализации») откладывается на более поздний Gate.
- Это не создаёт технического долга по отношению к уже принятым ADR — ADR-005/006/007/009 остаются валидными независимо от решения по C8.

### 8.13 Возможные метрики/критерии подтверждения понимания пространства

Ниже — потенциальные направления для критериев, а не установленный или согласованный framework:

- Точность извлечения сущностей и их атрибутов (тип объекта, приблизительные размеры/позиция) относительно размеченного ground truth набора помещений.
- Корректность ответов на структурированные запросы о пространстве (например, наличие прохода между двумя точками, объём свободного пола) относительно человеческой разметки.
- Наличие формального evaluation framework, отделяющего качество понимания от визуального качества генерации.

Данные пункты — предмет отдельной архитектурной/продуктовой проработки, если C8 будет выбран, а не готовый критерий приёмки.

## 9. Option Comparison: направления Gate 2

### Option A — Prompt Intelligence Evolution
**Состав:** C2 (Prompt Engine), C1 (Resolution Phase), C5 (Material/Color conflict), точечные улучшения generation pipeline (C6, C7).

- Продолжает уже принятую архитектурную линию (ADR-005/006/009).
- Governance-риск ограничен: только C2 требует нового ADR (раздел 5); C1/C4/C6/C7 — нет.
- Не вводит новую Platform Capability уровня Representation — остаётся в рамках Generation-слоя.
- Наименьший объём непроверенных архитектурных зависимостей.

### Option B — Semantic Spatial Intelligence Core (C8-centered)
**Состав:** C8 как основной кандидат Gate 2, C3 как его предварительный шаг.

- Прямо соответствует долгосрочному видению AI Interior Designer (раздел 8.4).
- Architecture Decision Required до начала любого implementation planning (раздел 8.7).
- Вводит новую, ранее не оценённую зависимость — Perception-слой (раздел 8.10).
- Потенциально требует пересмотра ADR-007.
- C1/C2/C4/C5/C6/C7 либо остаются независимыми, либо их связь с C8 не установлена — Gate 2 в этом варианте не закрывает их, они переносятся на другой Gate или трек.

### Option C — Hybrid Readiness Gate
**Состав:** C3 (ADR-004 gap review) выполняется первым как самостоятельный шаг; далее — отдельное Owner Decision о выборе между Option A и Option B (или их комбинацией) на основе результатов C3.

- Совместим с уже данной в разделе 6 рекомендацией («C3 рекомендуется рассмотреть первым»).
- Не требует немедленного выбора между Generation-центричным и Representation-центричным направлением.
- Результат C3 напрямую информирует решение по C8 (раздел 8.5, 8.6), снижая риск принятия решения о Gate 2 без данных.
- Откладывает, а не устраняет, необходимость выбора между Option A и Option B.

## 10. Owner Decision

**Решение:** ☑ Approved — Option C (Hybrid Readiness Gate)
**Дата:** 2026-07-09

**Strategic direction (принято):** Gate 2 должен двигаться в сторону Semantic Spatial Intelligence Core / Scene Understanding / Functional Scene Graph как основы для будущих AI Interior Designer capabilities.

**Explicit non-approval:** Финальный scope Gate 2 **не утверждён**. Утверждён только readiness-путь (шаги ниже), предшествующий утверждению scope.

### 10.2 Clarifications (зафиксированы Owner Decision)

- Prompt Intelligence Evolution (Option A) остаётся релевантным, но не определяет Gate 2 по умолчанию.
- C1 (Resolution Phase) остаётся доступным как downstream engineering package, но не является стратегическим центром Gate 2.
- C8 принят как потенциальный стратегический центр Gate 2, но **не утверждён для implementation**.
- Roadmap не создаётся.
- Implementation Package не создаётся.
- Планирование кода не начинается.
- Architecture Freeze не переоткрывается, если Architecture Decision Required явно не подтверждён и не утверждён отдельно.

### 10.3 Governance Status

**Architecture Decision Required по C8 не запущен формально.** Раздел 8.7 фиксирует это как предварительный архитектурный вывод (assessment finding), а не как утверждённое требование — формальное подтверждение возможно только после выполнения readiness-шагов 1–2 (сначала C3, затем формальная оценка C8). До завершения этих шагов ADR по C8 не готовится и Architecture Freeze не переоткрывается.
