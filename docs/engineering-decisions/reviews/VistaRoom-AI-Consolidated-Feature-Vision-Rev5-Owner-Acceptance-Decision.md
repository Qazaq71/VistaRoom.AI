## 3. Accepted product vision

The Project Owner accepts Revision 5 as the consolidated target product vision for VistaRoom AI.

VistaRoom AI is to be designed as the foundation of a complete:

```text
AI Interior Designer
+
Semantic Spatial Intelligence Platform
```

and not merely as an image-generation interface.

The accepted target capability set is represented by Sections A–S of Revision 5, including:

- perception and understanding of rooms;
- understanding of objects, materials and finishes;
- client dialogue and structured design brief formation;
- style intelligence;
- color and material intelligence;
- spatial planning and ergonomics;
- furniture and object planning;
- lighting design;
- controlled generation and editing;
- personalization;
- automated quality control;
- comparison of alternatives;
- materials, budget and procurement;
- reporting and project versioning;
- professional and B2B capabilities;
- cross-cutting foundations;
- residential room scope;
- proposed future product directions;
- bilingual localization.

Acceptance of these target capabilities as Product Feature Vision does not itself authorize their architecture, evaluation, implementation, deployment or commercial integration.

## 8. Controlled Learning & Continuous Improvement decision

The Project Owner confirms:

```text
Controlled Learning & Continuous Improvement:
OWNER-CONFIRMED FUTURE CROSS-CUTTING FOUNDATION
```

### 8.1 Current Spatial Perception cycle

The current Spatial Perception architecture cycle requires only:

```text
MINIMAL CONTROLLED LEARNING COMPATIBILITY FOUNDATION

learning-ready:
YES

learning-active:
NO
```

The required compatibility includes the ability to support:

- model version references;
- rule-set version references;
- contract version references;
- vocabulary version references;
- provider configuration version references;
- evidence;
- provenance;
- explicit traceability;
- reproducibility;
- future linkage between a result and user confirmation, rejection or correction;
- feedback semantic classification compatibility;
- mandatory separation of subjective preference/taste from factual error correction;
- consent and data-use eligibility extension points;
- immutable result and correction history;
- no-regression evaluation hooks;
- controlled rollout compatibility;
- rollback compatibility.

### 8.2 Personalization boundary

A user-specific taste profile is:

```text
PERSONALIZATION ONLY

NOT GLOBAL PRODUCTION MODEL/RULE LEARNING

NOT AN AUTOMATIC TRAINING SIGNAL
```

Subjective preference or taste must not be treated as equivalent to factual correction of a system error.

### 8.3 Future full architecture

The full Controlled Learning & Continuous Improvement Architecture remains a separate future governance cycle and must include:

- consent-based feedback collection;
- feedback classification;
- analytics;
- training eligibility;
- model and rule improvement;
- held-out evaluation;
- no-regression testing;
- versioned release;
- limited rollout;
- monitoring;
- rollback;
- auditability.

The following hard rule is accepted:

```text
User feedback must never directly modify production behavior.
```

Every future production model or rule change must be:

- prepared;
- versioned;
- tested;
- evaluated;
- separately authorized;
- observable;
- reversible.

This decision does not authorize the full Controlled Learning cycle or any learning-active production behavior.
